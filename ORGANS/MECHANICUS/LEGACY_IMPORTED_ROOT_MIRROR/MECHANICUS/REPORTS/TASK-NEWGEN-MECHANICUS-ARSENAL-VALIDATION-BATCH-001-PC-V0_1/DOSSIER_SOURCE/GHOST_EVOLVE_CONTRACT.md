# Ghost_Evolve Contract V0.1

## Definition

Ghost_Evolve is the IMPERIUM organ-agent evolution law.

Servitor must not only complete organ-involved tasks externally. Servitor enters the involved organ body, performs the organ's part from that organ's jurisdiction, and leaves reusable capabilities inside the organ.

## PASS condition

A task involving an organ is not fully successful unless the involved organ becomes stronger.

## Mechanicus interpretation

For a Mechanicus task, Servitor acts as Mechanicus hands and asks:

- What validation architecture should Mechanicus own?
- What tools/checkers should Mechanicus keep?
- What receipts prove trust?
- What Owner questions should Mechanicus ask?
- What scope exports should future Servitors receive?
- What recurring work can be turned into script/checker/playbook/schema?

## Required strengthening output

At least one reusable organ capability must be left behind:

- script
- checker
- schema
- registry update
- playbook
- owner question pattern
- report template
- receipt builder
- scope exporter

For this task, more than one is required because Mechanicus validation is a recurring core function.

## FAIL / WARN

FAIL_FOR_ORGAN_TRAINING if:
- task result is produced but no reusable Mechanicus capability is left;
- scripts are created but not registered/reported;
- receipts are missing;
- future Servitor cannot rerun the validation flow.

PASS_WITH_WARNINGS if:
- useful reusable capability exists but is partial;
- some scripts require follow-up;
- some validations remain missing due to absent tools.

## Not local LLM training

Ghost_Evolve is script-first, registry-first, receipt-first organ evolution. It is not local model training.
