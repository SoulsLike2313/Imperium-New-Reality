# Доктрина паспортов Mechanicus v0_2

**Документ:** `MECHANICUS_PASSPORT_DOCTRINE_V0_2_RU.md`
**Орган-владелец:** DOCTRINARIUM
**Связанная схема:** `imperium.tool_passport.v0_2`
**Связанные инструменты:** `passport_index_builder_v0_1`, `validator_rollout_v0_1`
**Версия доктрины:** v0_2
**Форма патча:** 7 (Heptagram)
**Статус:** действующая (с момента слияния v0.10.0 в master)

---

## 1. Назначение

Доктрина паспортов Mechanicus описывает **обязательный контракт** между любым инструментом Империума и системой управления знаниями о нём. Паспорт инструмента — это **самоописание**, которое позволяет инструменту быть найденным, провалидированным, прогнанным и учтённым без устной памяти владельца.

До v0_2 паспорта существовали как свободная форма: одни инструменты их имели, другие нет; одни декларировали валидаторы, другие нет; единого реестра не существовало. v0_2 закрывает эту дыру.

## 2. Принципы

1. **Каждый инструмент имеет паспорт.** Нет паспорта → нет инструмента. Любой `.py`/`.ts`/`.rs`/`.ps1` в `ORGANS/` который выполняет работу обязан иметь файл `*_tool_passport_v*.json` рядом с собой (как правило в подкаталоге `PASSPORTS/`).
2. **Паспорт декларирует валидаторы.** Минимум один. Без валидаторов инструмент считается «слепым» — он не доказывает свою работоспособность.
3. **Паспорт декларирует режим исполнения.** Один из {static, sim, paper, shadow, live}. Это говорит системе насколько опасно его прогонять автоматически.
4. **Паспорт декларирует owner-gating.** `owner_gated: true` означает что инструмент трогать без явной команды владельца запрещено даже в rollout. `false` означает безопасный авто-прогон.
5. **Паспорт ссылается на доктрину.** Поле `doctrine_ref` указывает на этот документ или другой релевантный закон.
6. **Паспорт указывает свой исполняемый файл.** Поле `tool_path` — относительный путь от корня репозитория. Если файла нет — паспорт «сирота».
7. **Реестр и ledger строятся автоматически.** Никакого ручного списка инструментов. `passport_index_builder` + `validator_rollout` дают каноническую картину.

## 3. Обязательные поля v0_2

| поле           | тип       | смысл                                                       |
|----------------|-----------|--------------------------------------------------------------|
| `schema_id`    | const     | строго `imperium.tool_passport.v0_2`                         |
| `tool_id`      | string    | snake_case, уникален в репо, паттерн `^[a-z][a-z0-9_]*$`     |
| `name`         | string    | человекочитаемое имя                                         |
| `version`      | string    | `^v[0-9]+(_[0-9]+)*$` (v0_1, v0_2, v1_0, ...)                |

| `lang`         | enum      | python / rust / typescript / javascript / powershell / ...   |
| `validators[]` | array     | минимум 1 валидатор с `name` и `exec`                        |
| `exec_mode`    | enum      | static / sim / paper / shadow / live                         |
| `owner_gated`  | boolean   | требуется ли явная команда владельца                         |

## 4. Желательные поля v0_2 (не обязательные, но настоятельно рекомендуемые)

- `build_system` — stdlib / cargo / npm / pnpm / pwsh / none
- `io_contract.{input_schema,output_schema,report_schema}` — пути к JSON Schema файлам в `ORGANS/MECHANICUS/SCHEMAS/`
- `observability.{stdout,metrics[],log_paths[]}` — как инструмент проявляет себя в наблюдении
- `capability_tags[]` — что умеет (indexer, scanner, validator, orchestrator, ...)
- `depends_on[]` — список `tool_id`'ов от которых зависит при запуске
- `tool_path` — путь к главному исполняемому файлу

## 5. Валидаторы

Каждый валидатор это пара `{ name, exec, timeout_seconds? }`. Поле `exec` — это shell-команда, в которой плейсхолдеры `{repo}` и `{tool_path}` будут заменены на абсолютный корень репозитория и относительный путь инструмента соответственно. По умолчанию timeout 10 секунд, максимум 300.

### Типовые валидаторы

- `syntax` — `python -m py_compile {tool_path}` или `cargo check` или `tsc --noEmit`
- `self_test` — запуск инструмента в режиме `--self-test` против синтетических данных
- `dry_run` — запуск против реального репо с `--dry-run --quiet`
- `lint` — проверка стиля (ruff / clippy / eslint)
- `contract_check` — валидация выхода против `output_schema`

### Дисциплина

Валидатор должен быть **детерминированным** и **быстрым** (≤5s в норме). Если валидатор требует сеть или внешние ресурсы — поднимать `timeout_seconds` и помечать в `capability_tags` тегом `network-required`.

## 6. Жизненный цикл инструмента

```
  add tool      →  add passport  →  add validators  →  passport_index_builder picks it up
                                                            ↓
  validator_rollout runs validators  →  rollout_ledger has receipt of pass/fail
                                                            ↓
  inquisition (manual)  ←  ledger has failed/timeout?  →   fix tool or retire it
```

Инструменты без паспорта или без валидаторов попадают в «serый список» × индекса и подлежат либо паспортизации либо отставке (см. Шаг 5 hygiene loop).

## 7. Receipts (квитанции)

Каждый прогон `validator_rollout` оставляет квитанцию в `ORGANS/MECHANICUS/TOOL_INTELLIGENCE/RECEIPTS/rollout_ledger_<timestamp>.json`. Квитанции — это **институциональная память**: они показывают что в момент времени T система знала о наборе инструментов S, прогнала валидаторы V, получила результаты R. История квитанций = история здоровья Mechanicus.

Правило хранения: квитанции **не удаляются**. Они могут архивироваться (rotation по дате), но не уничтожаются — это контрактный архив доказательств.

## 8. Связь с другими доктринами

- **TRINITY_PATCH_MANIFEST v0_2** — каждый патч-пак сам несёт паспортные изменения; паспорта новых инструментов входят в payload пака.
- **TRINITY_HYGIENE** — hygiene-сканер использует passport_index чтобы отличить «живые» инструменты от мусора.
- **WARP_DIFF_ROOM** — diff-room v0.9.9 показывает изменения паспортов между коммитами как отдельный класс артефактов.

## 9. Эталон формы Mechanicus (ответ на канон-вопрос владельца)

Инструмент считается **эталонным по Mechanicus** когда:

1. Имеет паспорт v0_2 со всеми обязательными полями
2. Имеет ≥2 валидатора (syntax + self_test/dry_run как минимум)
3. Все его валидаторы зелёные в последнем `rollout_ledger`
4. Указан `tool_path` и файл существует
5. Указан `doctrine_ref` на действующую доктрину
6. Имеет либо `io_contract` со схемами либо явное обоснование почему контракт не нужен
7. Наблюдаем через `observability.stdout` (не «none» если инструмент производит работу)

Эталоны формы являются образцом для всех последующих инструментов. Они же являются первыми кандидатами на портирование в Cockpit IDE в Шаге 6.

## 10. Поправки

Любое изменение этой доктрины требует:

1. Новый патч-пак формы 7 с обновлённой версией доктрины (`v0_3`, `v0_4`, ...)
2. Миграция всех существующих паспортов на новую схему
3. Прогон `validator_rollout` после миграции с результатом failed=0
