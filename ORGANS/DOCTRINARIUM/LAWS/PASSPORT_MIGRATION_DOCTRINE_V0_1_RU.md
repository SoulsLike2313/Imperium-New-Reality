# Доктрина миграции паспортов v0_1

**Документ:** `PASSPORT_MIGRATION_DOCTRINE_V0_1_RU.md`
**Орган-владелец:** DOCTRINARIUM
**Связанный инструмент:** `passport_migrator_v0_1`
**Родительская доктрина:** `MECHANICUS_PASSPORT_DOCTRINE_V0_2_RU.md`
**Версия:** v0_1
**Статус:** действующая с v0.10.1

---

## 1. Назначение

До v0.10.0 в репозитории существовали паспорта «v0_1» — без жёсткой схемы, непоследовательных имён, иногда без валидаторов. v0_2 ((`MECHANICUS_PASSPORT_DOCTRINE_V0_2_RU`)) наложила жёсткий контракт. Эта доктрина описывает как безопасно перевести легаси-паспорта на новый контракт.

## 2. Принципы

1. **Ничего не угадывать.** Мигратор НЕ изобретает валидаторы и НЕ вымышляет `owner_organ`. Если обязательное поле не выводится из существующих данных или пути — паспорт помечается `manual_migration_needed`.
2. **Бэкап всегда.** При `--apply` исходный файл паспорта копируется в `ORGANS/MECHANICUS/TOOL_INTELLIGENCE/PASSPORT_MIGRATOR/BACKUPS/<ts>/...` до записи новой версии.
3. **Dry-run первичен.** Сначала `--dry-run`, владелец смотрит ledger, только потом `--apply`. Инструмент owner_gated=true.
4. **Каноническое имя обязательно.** Паспорты пишутся по формату `<tool_id>_tool_passport_<version>.json` в подкаталоге `PASSPORTS/` рядом с инструментом. Если исходный файл в другом месте — он переписывается в канон, старый удаляется, бэкап остаётся.
5. **Ledger обязателен.** Каждый прогон (и dry-run, и apply) пишет `imperium.passport_migration_ledger.v0_1` в указанный `--out`.

## 3. Правила вывода полей v0_2

| поле v0_2     | правило вывода |
|---------------|----------------|
| `schema_id`   | всегда `imperium.tool_passport.v0_2` |
| `tool_id`     | (a) из существующего паспорта если валиден; (b) из имени файла если подходит паттерн; (c) из stem файла после нормализации; иначе `missing` |
| `name`        | из паспорта; иначе из `tool_id` через Title Case |
| `version`     | из паспорта если валиден; иначе «v0_1» |
| `owner_organ` | (a) из паспорта если в перечне; (b) из пути файла (`ORGANS/MECHANICUS/...` → MECHANICUS); иначе `missing` |
| `lang`        | (a) из паспорта; (b) из расширения `tool_path`; (c) из расширения соседнего исполняемого файла; иначе `missing` |
| `validators`  | только из паспорта; НИЧЕГО не выдумывается; если нет — `missing` |
| `exec_mode`   | из паспорта если в перечне; иначе «static» |
| `owner_gated` | из паспорта если bool; иначе false |
| `tool_path`   | из паспорта; иначе первый исполняемый файл в соседней директории (.py/.rs/.ts/.ps1) |
| `updated_at`  | всегда текущее UTC время миграции |

## 4. Статусы в ledger

- `skipped_already_v0_2` — паспорт уже на v0_2, не трогаем.
- `planned` (только в dry-run) — может быть мигрирован без потерь.
- `applied` (только в apply) — мигрирован, бэкап сохранён.
- `manual_migration_needed` — не хватает обязательных полей; владелец редактирует ручной.
- `parse_error` — файл невалидный JSON.

## 5. Порядок работы владельца

```
1.  python passport_migrator_v0_1.py --repo $HRepo --dry-run --out E:\_LOCAL_HANDOFF\MIGRATION\<ts>.json
2.  Открыть ledger, проверить manual_migration_needed[] и планируемые изменения
3a. Если manual = 0 и planned выглядят правильно:
    python passport_migrator_v0_1.py --repo $HRepo --apply --out E:\_LOCAL_HANDOFF\MIGRATION\<ts>_applied.json
3b. Если manual > 0:
    открыть каждый такой паспорт вручную, добавить недостающие поля по рекомендациям ledger, повторить dry-run
4.  После apply: прогнать passport_index_builder + validator_rollout, убедиться что non_compliant=0
```

## 6. Безопасность

- `--apply` без визуального обзора сначала dry-run — нарушение доктрины.
- Бэкапы в `BACKUPS/<ts>/` НИКОГДА не удаляются автоматически. Чистка выполняется hygiene-сканером (`v0.9.8`) по возрасту/политике.
- После миграции обязателен второй прогон `validator_rollout` (не dry-run) чтобы подтвердить работоспособность валидаторов теперь-v0_2-паспортов.

## 7. Ограничения v0_1

- Не выполняет подстройку схем-сирот (`io_contract` без файла по пути) — только переносит как есть.
- Не переименовывает исполняемые файлы инструментов — только паспорта.
- Не редактирует код валидаторов — если легаси-validator использует неподдерживаемый плейсхолдер, это всплывёт в последующем rollout как fail.

## 8. Поправки

Любое расширение логики мигратора требует новой версии (`passport_migrator_v0_2`, `v0_3`, ...) с обновлённым паспортом и этой доктрины.
