# ПЛЕЙБУК: САМЫЙ ТОЧНЫЙ И САМЫЙ ДЕШЁВЫЙ ОТВЕТ ОТ LLM

> Свод практик из открытых источников (Anthropic, OpenAI, Google, arXiv, инженерные блоги 2025–2026),
> адаптированных под архитектуру Империума. Статус: CANDIDATE_NOT_CANON.

## 0. Главный закон

**Токены — это не «размер», это «бюджет».** Исследования Anthropic и телеметрия Praetorian: **объём токенов объясняет ~80% разброса качества в агентных задачах**. Чем больше мусора в контексте — тем хуже и дороже. Цель — максимум сигнала на минимуме токенов.

---

## 1. ДЕШЕВИЗНА: как снизить аппетит системы

### 1.1. Prompt caching (кэш префикса) — самый быстрый выигрыш
- Провайдеры кэшируют повторяющийся префикс (системный промпт, инструкции, большие неизменные данные).
- **Anthropic:** чтение из кэша — ~10% от цены свежих токенов ($0.30 вместо $3.00/M), до **−90% цены** и −85% латентности. Минимум блока — 1024 токена. Первый вызов +25%, дальше −90%.
- **OpenAI:** авто-кэш по умолчанию, −50% при попадании.
- **Правило Империума:** стабильный системный префикс (контракт Сервитора) — ПЕРВЫМ, переменная часть — ПОСЛЕ. Реализовано в `metaos_orchestrator.py` (`cached_prefix_tokens`).

### 1.2. Обрезка и сжатие контекста
- **Системный промпт-раздув — тихий убийца.** 1200-токенный промпт, урезанный до 400, экономит 800 токенов на КАЖДОМ вызове.
- **Context pruning (Provence и аналоги):** убираем предложения, не относящиеся к вопросу, ПЕРЕД отправкой.
- **Prompt compression (LLMLingua):** маленькая модель выкидывает неважные токены, до 20x сжатия при ~95% сохранении качества.
- **Lost-in-the-middle:** важное кладём в начало/конец, не в середину. Реализовано в `ContextWindow.materialize()`.

### 1.3. Кэш ответов (semantic / exact)
- **Многоуровневый кэш:** exact-match (суб-мс) → semantic (похожие запросы) → inference. ~31% запросов семантически похожи; semantic-кэш режет до −73% расхода. Кэш-попадание вообще обходит вызов модели.

### 1.4. Формат вывода
- **JSON стоит ~2x токенов против TSV.** Для внутренних вызовов — компактные форматы (TSV/TOON) в теле промпта, JSON — только на границе API.
- **Минимальный структурный вывод** через системный промпт режет до 60–80% токенов для backend-вызовов.

## 2. ТОЧНОСТЬ: как повысить качество без перерасхода

### 2.1. Structured output с валидацией — самый дешёвый фикс
- OpenAI Structured Outputs: соответствие схеме **<40% → 100%** при strict JSON Schema.
- Финансовая команда: ошибки 5% → <0.3% после схемы. Цикл валидации: точность шагового воркфлоу 10% → 70%.
- **В Империуме:** это и есть гейт Администратума — нет валидной формы = нет выпуска.

### 2.2. Self-consistency — осторожно, дорого
- Мажоритарное голосование по нескольким путям — +5–25% на reasoning (GSM8K +17.9%). НО это ×5–10 токенов.
- **Вывод 2025–2026:** на современных моделях выгода падает (diminishing returns). Лучше: **confidence-aware / early-stopping SC** — −40–65% выборок при том же качестве. Применять ТОЛЬКО на критичных задачах.

### 2.3. Reasoning effort вместо ансамбля
- Стратегический выбор модели + уровень reasoning эффективнее ансамблирования. Повышать reasoning только там, где он реально нужен.

## 3. ОРКЕСТРАЦИЯ: routing + cascade

| Паттерн | Как работает | Когда |
|--------|--------------|-------|
| **Routing** | Одно решение до выполнения: классификация запроса → 1 модель | Известна сложность заранее |
| **Cascade** | Сначала дешёвая; если confidence < порога — эскалация выше | Неопределённая сложность |

- Многомодельная оркестрация режет production-расход до ~60% (2026 отраслевые отчёты).
- Дешёвые/локальные модели — 10x–100x дешевле, но слабее на reasoning → идеальны для простых задач в каскаде.
- **В Империуме:** `MetaOSOrchestrator` — auto-routing + cascade, жёсткий token_budget.

## 4. SCRIPT-FIRST: перенос ручного труда в код (органы)

Главный рычаг экономии: **не заставлять LLM делать то, что может детерминированный код.**

- **Compiled AI / мета-тулы (arXiv, Microsoft Conductor):** повторяющиеся цепочки вызовов сворачиваются в один детерминированный «мета-тул» → меньше шагов reasoning, меньше сбоев.
- **Thin agent:** LLM выбирает способность и строит план, а валидация/исполнение — детерминированны. Бизнес-правила — в коде, не в промпте.
- **Routing-агент:** лёгкая классификация на входе сокращает лишние проходы ретрива.

Соответствие твоей модели:
- **Ядро/органы** держат постоянный труд: отбор контекста, правила, ретрай, валидация, летопись.
- **Сервитор** — одноразовый исполнитель одной задачи на отобранном контексте, без лишних чтений и рецептов.

## 5. ЧТО БЫСТРО ИНТЕГРИРОВАТЬ В ИМПЕРИУМ (по убыванию ROI)

1. **Prompt caching** на системном префиксе Сервитора — выигрыш за 1 день, до −90%.
2. **Жёсткий token_budget на задачу** + отказ при превышении (`metaos_orchestrator.py`).
3. **Отбор контекста органом** (top-N релевантных фрагментов) — `servitor_runtime.ContextWindow`.
4. **Routing/cascade** дешёвая→дорогая модель.
5. **Bundle-report gate Администратума** — строгая форма выпуска (`administratum_bundle_gate.py`).
6. **Метрика расхода** (meter-before-manage): лог токенов/цены на задачу — без измерения нет оптимизации.
7. **Кэш ответов (exact → semantic)** — позже, когда появится повторяющийся трафик.

---

## 6. Связь с WARP и гейтами

- Уровни доказательств **E1..E6** из WARP-доктрины используются в bundle-репорте (минимум E3).
- Администратум-гейт — это частный случай WARP-GATE: fake-green (pass без доказательств) → не выпускается.
- Вся грязь оркестрации (рантаймы, попытки) живёт в WARP-зоне, ядро чистое.

---

## 7. Честные ограничения

- Цифры экономии (−90%, −60%, −73%) — из внешних кейсов; на твоём трафике надо замерить.
- Кодовые примеры — референс-скелеты со стабами вместо реальных LLM API. Подключи свой backend в `runner`.
- Токен-оценка (len//4) — грубая; для точности подключи реальный tokenizer.
