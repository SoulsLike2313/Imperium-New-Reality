# WARP — доктрина горячей зоны разработки

## Что такое WARP одним абзацем
WARP — это горячая зона разработки внутри IDE. Любая работа — правим ли мы
ядро, строим ли силы для Империума или собираем стороннее ПО — идёт ТОЛЬКО
внутри WARP. Ядро при этом не ломается и не загрязняется: оно живёт отдельно
и только читается. Продукт покидает WARP только через шлюз (GATE). Не смогли
сделать хорошо и не переделали — просто не выпустили. Ядро чистое.

## Трёхслойная модель
```
        ┌──────────────────────────────────┐
        │             WARP  (воркрум)            │   вся грязь, рантаймы,
        │  runtimes · products · dev-отходы      │   дифы, метрики, стадии
        │  diffs · stages · criteria · metrics    │
        └───────────────▲────────────────┘
                        │ GATE (release | hold | discard)
        ┌───────────────┴────────────────┐
        │         IDE  (оркестратор)            │   садится поверх ядра,
        │  ресурсы · кнопка WARP · авто-хук       │   управляет ресурсами
        └───────────────┴────────────────┐
        │         KERNEL  (ядро)                 │   read-only baseline,
        │  органы · законы · канон              │   меняется ТОЛЬКО через
        │                                       │   gated release plan
        └───────────────────────────────┘
```

- **KERNEL (ядро).** Защищённое сердце: органы, законы, канон. Для WARP это
  read-only baseline. Никто не пишет в ядро напрямую.
- **IDE (оркестратор).** Садится поверх ядра, раздаёт ресурсы, держит кнопку
  WARP и авто-хук старта задачи.
- **WARP (воркрум).** Вся работа, все рантаймы, все продукты и отходы. Здесь
  видны дифы, стадии, план, критерии, метрики и кто/что используется.

## Кто работает внутри WARP
Когда стартует задача, ВСЁ регистрируется как участники сессии и работает
внутри её границ: органы, Сервиторы, чтение ядра, API, CLI, инструменты.
У каждого участника фиксируется режим: `read` / `dry-run` / `real`.

## Три вида работы (kind)
- **CORE_CHANGE** — правка самого ядра. Самый жёсткий режим: релиз только с
  одобрением владельца.
- **IMPERIUM_FORCE** — строим силы/органы для Империума.
- **THIRD_PARTY** — стороннее ПО/продукт. Живёт в WARP, в ядро вообще может не
  промоутиться — выходит как самостоятельный артефакт.

## Почему это «жёсткая инженерная форма»
1. **Изоляция физическая.** Запись идёт только в `WARP/runtime/<session>/artifacts/`.
   В ядро нет пути записи вообще.
2. **Шлюз обязателен.** `GATE` считает критерии+метрики и даёт вердикт
   RELEASE / HOLD / DISCARD. Нет RELEASE — нет выхода.
3. **Анти fake-green.** Если метрика заявляет «pass», но доказательство ниже
   порога (E1/E2) — вердикт DISCARD. Нельзя «покрасить зелёным».
4. **История неизменяема.** `events.jsonl` — append-only.
5. **Ядро-релиз под одобрением.** Продвижение в CORE-зону требует одобрения
   владельца — release_manifest выходит со статусом PENDING_OWNER_APPROVAL.
