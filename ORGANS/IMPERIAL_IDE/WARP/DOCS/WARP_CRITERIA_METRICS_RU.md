# Критерии и метрики WARP

## Лестница доказательств (evidence ladder)
| Уровень | Смысл | Пример |
|--------|-------|--------|
| E1 | существует | файл/структура созданы |
| E2 | само-декларация | «я сделал» без пруфа (НЕ доверять) |
| E3 | выполнено | реальный запуск с логом |
| E4 | стабильный повтор | проходит дважды/реплей |
| E5 | аудит | проверено независимо/дифы одобрены |
| E6 | owner UX | владелец видел результат в IDE |

Порог по умолчанию: **E3**. Меньше — считается недоказанным.

## Базовые метрики (можно расширять)
| Метрика | Что показывает |
|---------|----------------|
| `builds` | код собирается/компилируется |
| `tests_pass` | тесты проходят |
| `no_core_pollution` | нет прямых записей в ядро |
| `diffs_reviewed` | дифы просмотрены |
| `runtime_clean` | рантайм без ошибок/утечек |
| `owner_ux_ok` | владелец подтвердил |

## Как записать метрику
```python
session.record_metric("tests_pass", value=True, evidence_level="E4",
                      note="pytest 14 passed, replay ok", source="pytest.log")
```

## Как шлюз считает вердикт
1. Для каждого критерия берётся его метрика.
2. Нет метрики или value!=expect у обязательного → HOLD.
3. value==expect, но evidence < min_evidence → fake-green → DISCARD.
4. Всё обязательное выполнено и доказано → RELEASE.
