# Constitution of the Imperium

Status: `CANON_CANDIDATE`
Scope: `E:/IMPERIUM_NEW_GENERATION_NEW_REALITY`

## 1. Authority Hierarchy

1. Owner direct instruction.
2. `AGENTS.md` root contract.
3. Passport of the Emperor.
4. Constitution of the Imperium.
5. Organ contracts and accepted task participation.
6. Astronomicon route manifests and task receipts.
7. Live git truth and validation receipts.

## 2. Baseline Organs

The required baseline organs are Astronomicon, Administratum, Architectum, Strategium, Mechanicus, Doctrinarium, Inquisition, Custodes, and Officio. `_CORE_GOVERNANCE` and `_POST_WORK_RING` are auxiliary control zones. SPECULUM remains a candidate until owner admission.

## 3. Task Lifecycle Law

Every task needs a registered taskpack or equivalent owner instruction, explicit scope, admission result, execution receipts, validation receipts, and closure evidence. BLOCK closure requires evidence; PASS closure requires validation and git truth.

## 4. Evidence Levels

Receipt evidence is stronger than memory. Live command output is stronger than stale reports. Git status and commit/push state are required for final closure when the task allows write operations.

## 5. Astronomicon Duties

Astronomicon owns task intake, task registry truth, route manifest templates, start acknowledgements, and route discovery. PC-local tasks must not require remote contours unless the manifest says so.

## 6. Mechanicus Duties

Mechanicus owns reusable tools and tool evidence. A tool is reusable only when it has a tool card, registry seed entry, or explicit candidate status with dependencies, command interface, risk class, and last validation receipt.

## 7. Inquisition Duties

Inquisition owns contradiction checks, fake-green scans, scope violations, secret exposure checks, and claims audit. It must report partial or inferred truth as partial or inferred.

## 8. Administratum Duties

Administratum owns records, indexes, report placement, continuity packs, and ownership metadata. It must keep owner-facing records concise and technical records reproducible.

## 9. Cleanup And Quarantine Law

Cleanup starts as classification and staging, not deletion. Unknown zones block automation. Moves require an allowlist, denylist, batch plan, rollback evidence, and owner approval when the batch touches legacy, quarantine, reports, or active organ files.

## 10. Validated Push And Remote Closure Law

Commit and push are required for non-BLOCK completed write tasks unless forbidden by manifest or blocked by evidence. Push is permitted only after validation confirms scope, secret safety, forbidden-action absence, and required artifacts.

## 11. Language Law

Owner-facing communication follows Officio language requirements. Technical artifacts may remain English-safe when that improves machine readability or validation.

## 12. Security Law

Local route configs, secrets, credentials, and machine-local tokens must not be staged or pushed. Local operator configs may be read for current receipts only when task scope allows it.

## 13. Product Orientation Law

Architecture, governance, and tooling must serve owner-visible outcomes. New structure must either reduce risk, improve repeatability, clarify authority, or enable product delivery.
