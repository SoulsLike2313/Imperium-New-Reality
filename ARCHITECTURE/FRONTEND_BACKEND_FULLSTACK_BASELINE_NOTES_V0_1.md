# Frontend/Backend Full-Stack Baseline Notes V0.1

## Why this note exists
Owner reference images provide beginner structure examples.
They are useful as baseline orientation, but insufficient for IMPERIUM execution quality.

## Baseline educational shape
Common structure:
- `frontend/src/components|pages|services|utils`
- `backend/controllers|routes|services|models`
- optional `database/`

## Why baseline alone is insufficient for IMPERIUM
- no domain ownership model;
- no truth/evidence corridor;
- no role admission and gate discipline;
- no no-fake-green control;
- no skill passport layer;
- no visual topology and address system;
- no validator-driven readiness criteria.

## IMPERIUM adaptation principles
1. Separate apps from shared packages and backend services.
2. Keep contracts/schemas independent from runtime implementation.
3. Treat receipts/reports/screenshots as first-class evidence artifacts.
4. Require role and gate acknowledgements before mutable work.
5. Use skill passports to reduce generic LLM behavior drift.
6. Isolate runtime outputs from canonical source.

## Frontend architecture baseline for New Generation
- design tokens and motion are package assets, not app-local defaults;
- UI primitives are separate from domain widgets;
- visual topology addresses map UI units to truth sources;
- reduced-motion/accessibility constraints are part of skill protocol.

## Backend architecture baseline for New Generation
- API and adapters are contract-bound;
- route/task/evidence registries are explicit services;
- no fake live backend claim without verifiable endpoints and receipts;
- persistence candidates are declared before implementation commitments.

## Operational guardrails
- no broad toolchain adoption in this task;
- no ecosystem installation claims without admission evidence;
- architecture readiness is partial until migration tasks execute.
